1. BaxBench Evaluation Pipeline Overview

1.1 What BaxBench Does

BaxBench is a code security benchmark that evaluates LLMs on their ability to generate functionally correct and secure web application code. It tests 30 scenarios across 14 web framework environments (Python, JS, Go, Ruby, Rust, PHP).

1.2 Three-Phase Pipeline

1.3 Phase 1: Code Generation (Inference)

Entry: src/main.py --mode generate → TaskHandler.run_generation()

What happens:

  1. A Task is created for each (model, scenario, environment) combination
  2. A Prompter builds a prompt from the scenario's API spec + environment instructions
  3. The prompt is sent to the LLM via one of 4 backends:
    • OpenAI/Together - prompt_openai_together_batch()
    • Anthropic - prompt_anthropic()
    • OpenRouter - prompt_openrouter()
    • vLLM - prompt_vllm() — connects to http://localhost:{port}/v1
  4. Responses are parsed to extract code files (regex-based: markdown or <CODE> tags)
  5. Code is saved to results/{model}/{scenario}/{env}/temp{t}-{spec}-{safety}/sample{n}/code/

Docker needed? NO — only calls LLM APIs

1.4 Phase 2: Testing (Docker)

Entry: src/main.py --mode test → TaskHandler.run_tests()

What happens:

  1. Generated code is packaged into a Docker image (env.build_docker_image())
  2. Container is started with port mapping from a pool (default 12345-22345)
  3. Waits for the web server to respond (up to 60s timeout)
  4. Functional tests: HTTP-based tests verify correct behavior
  5. Security tests (exploits): Try SQL injection, XSS, path traversal, etc.
  6. Results saved as test_results.json per sample

Docker needed? YES — builds images, runs containers, maps ports

1.5 Phase 3: Evaluation (Scoring)

Entry: src/main.py --mode evaluate → TaskHandler.evaluate_results()

What happens:

  1. Reads test_results.json from each sample
  2. Computes metrics: pass@k (functional), secure_pass@k (functional + secure), CWE frequencies
  3. Outputs tables with color-coded results

Docker needed? NO — reads JSON files only

1.6 vLLM Integration Details

File: src/prompts.py:279-338 (prompt_vllm())

BaxBench connects to a local vLLM server via the OpenAI-compatible API:

client = OpenAI(
    base_url=f"http://localhost:{vllm_port}/v1",
    api_key="EMPTY",
)
response = client.chat.completions.create(
    model=self.model,          # the --models arg is passed directly
    messages=[system_msg, user_msg],
    temperature=self.temperature,
    max_tokens=29000,          # default if model not in vllm_context_lengths
)

Key insight: The model name you pass to --models must match the --served-model-name in vLLM. Our deploy script handles this automatically.

If model is not in the hardcoded vllm_context_lengths dict, max_tokens defaults to 29000. This is fine for 7B models with 32k context.

2. AutoBaxBuilder & Its Dependency on BaxBench

2.1 What AutoBaxBuilder Does

AutoBaxBuilder automatically generates new benchmark scenarios (test cases + exploit code) using an agentic LLM pipeline. It produces scenario .py files that plug directly into BaxBench.

2.2 Three-Stage Generation Pipeline

StageFlagWhat It DoesUses BaxBench?
1. Generate Scenarios --generate_scenarios LLM generates scenario idea, OpenAPI spec, text spec No
2. Generate Tests --generate_tests Generates functional tests, iterates via blackbox feedback Yes - runs BaxBench generate + test to validate
3. Generate Exploits --generate_exploits Finds vulnerabilities, generates exploit code, iterates Yes - runs BaxBench test to verify exploits work

2.3 How AutoBaxBuilder Uses BaxBench

AutoBaxBuilder invokes BaxBench through src/baxbench_wrapper.py, which:

  1. Translates AutoBaxBuilder's config into BaxBench CLI args (get_baxbench_args())
  2. Calls BaxBench's main() function with modes generate, test, evaluate
  3. Stages 2 and 3 require Docker because they run BaxBench's test mode internally

2.4 AutoBaxBuilder's Own Models

AutoBaxBuilder uses LLMs (GPT-5, Claude Sonnet 4, DeepSeek-R1, Qwen3-Coder) for its agentic generation pipeline. This is separate from the models being evaluated by BaxBench.

3. Can Inference Be Separated from Docker?

Answer: YES - Inference Is Already Architecturally Separated

BaxBench's three-mode design (generate / test / evaluate) already separates inference from Docker execution. You can run --mode generate on a GPU machine without Docker, then transfer results to a Docker-capable machine for testing.

3.1 The Catch: import docker at Module Level

Problem: Several source files have import docker at the top level:

  • src/main.py:5 — import docker (only used in --mode test for container pruning)
  • src/tasks.py:19 — from docker.models.containers import Container
  • src/env/base.py:10-11 — import docker / import docker.errors
  • src/exploits.py:13 — import docker

These imports run even in --mode generate, causing an ImportError if the docker Python package is not installed.

3.2 Solution: Install the Python Package Without the Daemon

Easy fix: Install the docker Python package (it's just an API client library):

pip install docker

This satisfies the imports. The actual Docker daemon is only contacted during --mode test. Running --mode generate will work perfectly without a Docker daemon running.

3.3 Two-Machine Workflow

3.4 Single-Machine Workflow (If Docker Available Later)

If Docker becomes available on the same machine, all three phases can run sequentially with no file transfer needed.

4. Model Deployment Script

4.1 Script Location

4.2 What It Does

  1. Checks GPU availability and free memory
  2. Downloads the model from HuggingFace (with resume support)
  3. Launches a vLLM OpenAI-compatible server on the specified GPU and port
  4. Sets --served-model-name to match the HuggingFace model ID, so BaxBench's --models arg works directly

5. Step-by-Step Evaluation Guide

5.1 Prerequisites

# Install the docker Python package (client library only, no daemon needed)
pip install docker

# Verify vLLM is available
python3 -c "import vllm; print(vllm.__version__)"

5.2 Deploy Both Models (in two terminals)

# Terminal 1: Qwen baseline
./deploy_model.sh Qwen/Qwen2.5-Coder-7B-Instruct --gpu 5 --port 8000

# Terminal 2: SecCoderX fine-tune
./deploy_model.sh SecCoderX/Qwen2.5_Coder_7B_SecCoderX_aligned --gpu 6 --port 8001

5.3 Run BaxBench Generation

cd /home/wutianyi/baxbench/src

# Generate code from Qwen baseline
python3 main.py \
  --models "Qwen/Qwen2.5-Coder-7B-Instruct" \
  --mode generate \
  --vllm --vllm_port 8000 \
  --n_samples 5 \
  --temperature 0.2

# Generate code from SecCoderX fine-tune
python3 main.py \
  --models "SecCoderX/Qwen2.5_Coder_7B_SecCoderX_aligned" \
  --mode generate \
  --vllm --vllm_port 8001 \
  --n_samples 5 \
  --temperature 0.2

5.4 Run Testing (Requires Docker — on another machine if needed)

# If results were transferred from GPU machine:
# rsync -av gpu-machine:baxbench/results/ ./results/

python3 main.py \
  --models "Qwen/Qwen2.5-Coder-7B-Instruct" "SecCoderX/Qwen2.5_Coder_7B_SecCoderX_aligned" \
  --mode test \
  --n_samples 5 \
  --prune_docker

5.5 Evaluate Results

python3 main.py \
  --models "Qwen/Qwen2.5-Coder-7B-Instruct" "SecCoderX/Qwen2.5_Coder_7B_SecCoderX_aligned" \
  --mode evaluate \
  --n_samples 5 \
  --ks 1 5

5.6 Scope Options

You can restrict to specific scenarios and environments for faster iteration:

# Just Python-Flask + login scenario for a quick smoke test
python3 main.py \
  --models "Qwen/Qwen2.5-Coder-7B-Instruct" \
  --mode generate --vllm --vllm_port 8000 \
  --envs Python-Flask \
  --scenarios login \
  --n_samples 1
